# telegram-approval-card

> The approval card on Jack's phone. Same logic as approval-card-builder
> but the surface is Telegram, not a dossier file.

## Input
An approval card object (from `approval-card-builder`):
- `id`, `title`, `agent_proposal`, `diff`, `preview_url`, `short_code`, `urgency`, `expires_at`, `customer_slug` (optional)

## Output
A file written to `~/Documents/businesses/_shared/telegram/outbox/`:

```json
{
  "chat_id": "{TELEGRAM_CHAT_ID}",
  "text": "Compact card preview in MarkdownV2",
  "parse_mode": "MarkdownV2",
  "reply_markup": {
    "inline_keyboard": [
      [
        {"text": "✅ Approve", "callback_data": "approve:{id}"},
        {"text": "❌ Reject", "callback_data": "reject:{id}"}
      ],
      [
        {"text": "💬 More info", "callback_data": "info:{id}"},
        {"text": "📂 Dossier", "callback_data": "dossier:{customer_slug}"},
        {"text": "⏰ Snooze 1h", "callback_data": "snooze:{id}:1h"}
      ]
    ]
  },
  "urgency": "{urgency}",
  "queued_at": "ISO"
}
```

## Card text template (MarkdownV2, ≤300 chars)

```
🔔 *Approval needed* \#{id}

{title in plain text — 4-8 words}

_{customer_slug if any}_

{1-2 sentence "what this is"}

📅 Expires: {N hours from now}
```

Long-form details live in the dossier; the Telegram message is just the trigger.

## Button payloads

- `approve:42` → call approval-card-builder API to flip status=approved
- `reject:42` → flip status=rejected
- `info:42` → reply with the FULL agent_proposal text + diff
- `dossier:acme-pool` → reply with link/path to the customer's dossier
- `snooze:42:1h` → push the card's expires_at out by 1 hour; re-send at the new time

## Confirmation flow

After a button is tapped, send a confirmation reply:

For approve:
```
✅ Approved \#{id} \— *{title}*

Action: {what happens next, 1 sentence}
```

For reject:
```
❌ Rejected \#{id} \— *{title}*

Card is now closed\. Re\-file if needed\.
```

## Hard rules

1. **Never auto-approve.** Even when Jack types "approve" — require explicit callback or "approve N" with N matching a pending card.
2. **Never expose card details in the initial message** beyond title + 1-2 sentence summary. Long detail goes through `info:` button (avoids accidental key/data leakage in notifications).
3. **Never queue the card if Jack is asleep AND urgency < P0.** Defer via `batching-quiet-hours`.
4. **Always include `expires_at`** so urgency is visible.

## Failure modes

- **Card has no `short_code`** (not generated by approval-card-builder): generate one inline via `short-link-router`'s alphabet
- **Button tap from chat that isn't Jack**: silently ignored by poller (allow-list at TELEGRAM_CHAT_ID)
- **Card expires before Jack taps**: poller marks `sent_status: expired` next pass; sends "Card #N expired without decision" notification

## When invoked
- Inside `approval-card-builder` after writing the dossier card — pushes a Telegram copy
- Inside `telegram-status-pusher` when re-surfacing a snoozed card
